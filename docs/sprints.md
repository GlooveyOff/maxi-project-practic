# Отчёты по спринтам

Каждый спринт длился 2 недели. В конце каждого собирали ретроспективу,
фиксировали что получилось, что нет и что забрали в следующий.

## Спринт 1 — каркас приложения

**Цель:** поднять минимально работающий бэкенд и фронт, договориться
о ролях и пользовательских историях.

**Сделано:**
- Описаны 3 роли (`guest`, `employee`, `admin`) и 10 user stories
  ([docs/user_stories.md](user_stories.md)).
- Подняли FastAPI с эндпоинтами `/api/health` и заглушкой каталога.
- Сделали минимальный фронт на React + Vite (одна страница со списком).
- Написали 1 тест pytest для `/api/health`.

**Что не успели:** авторизацию, БД (работали на in-memory заглушках).

## Спринт 2 — база данных и реальные модели

**Цель:** перевести проект на PostgreSQL, описать сущности, настроить
миграции и тестовое окружение.

**Сделано:**
- Подняли PostgreSQL 15 в Docker (`docker-compose.yml`).
- Описали 4 сущности (`User`, `OilField`, `Well`, `MaintenanceRequest`)
  через SQLAlchemy 2.0.
- Настроили Alembic, написали миграцию `0001_initial`.
- Перевели роутеры `/api/fields` и `/api/wells` на работу с БД.
- Подняли отдельную тестовую БД на порту 5433.
- Написали 5 тестов на auth/fields/wells.

**Что не успели:** красивый UI, фильтры на фронте.

## Спринт 3 — git-процесс и качество

**Цель:** научиться работать через PR-flow, договориться о ревью.

**Сделано:**
- Включили *Branch protection* на `main` (запрет прямого push,
  обязательный PR + 1 approve).
- Добавили всех участников команды в репозиторий как коллабораторов.
- Прошли полный цикл feature-ветка → PR → review → merge как минимум
  по одному разу каждый.
- Завели шаблон описания PR («что сделано» / «как проверить»).
- Покрыли тестами доступ без авторизации (401) и unknown route (404).

**Что не успели:** CI с автозапуском тестов на PR.

## Спринт 4 — расширение функциональности и подготовка к защите

**Цель:** довести проект до состояния, в котором его не стыдно
показывать, и подготовить демонстрацию.

**Сделано:**
- Добавили новую сущность `Brigade` (бригады обслуживания) с
  миграцией `0002` и связью с заявками через `brigade_id`.
- Расширили API: эндпоинты `/api/brigades`, `/api/users` (для админа),
  `/api/stats/overview`.
- На фронте добавили страницы «Бригады» и «Статистика» (дашборд).
- Написали README со всеми обязательными блоками: описание, стек,
  структура, ENV, запуск.
- Подготовили GitHub Actions для прогона `pytest` на каждый push.
- Завернули backend и frontend в Dockerfile'ы.
- Написали сценарий защиты ([docs/presentation.md](presentation.md)).

## Ретроспектива по итогам курса

**Что получилось:**
- Уложились в дедлайны по всем 4 спринтам.
- Удалось разделить роли так, что у каждого был свой кусок и
  никто не блокировал других.
- PR-flow к концу курса стал привычкой — никто уже не пушил в `main`
  напрямую.

**Что было сложно:**
- Сначала путались с миграциями Alembic — несколько раз пришлось
  ронять локальные БД и накатывать заново.
- Не сразу разобрались с CORS, когда фронт начал ходить в API.
- Долго подбирали схему авторизации (JWT vs сессии) — в итоге выбрали
  JWT ради простоты на демо.

**Чему научились:**
- Работе через PR с обязательным код-ревью.
- Разделению сущностей между backend, БД и фронтом без дублирования.
- Тестированию HTTP-эндпоинтов с подменой зависимостей FastAPI.
- Запуску всего стека одной командой через `docker compose`.
