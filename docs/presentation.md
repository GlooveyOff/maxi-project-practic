## Текст выступления

### 1. Краткое представление (≈1,5 мин)

> «Наш проект называется «Нефтегаз — корпоративный портал». Это веб-приложение
> для нефтепромышленной компании: ведение справочника месторождений и скважин,
> создание заявок на обслуживание и разграничение доступа по ролям.
> Frontend написан на React + Vite, backend — на FastAPI и SQLAlchemy,
> база данных — PostgreSQL, авторизация через JWT.
> Проект покрыт интеграционными тестами с отдельной тестовой БД
> (pytest + httpx).»

### 2. Демонстрация (≈2 мин)

> «Запускаю проект: backend на порту 8000, frontend на 5173, PostgreSQL
> поднят в Docker. Открываю главную страницу — видно общую информацию о
> компании. Перехожу в «Месторождения»: список открытых месторождений с
> фильтрацией по названию.
>
> Захожу под администратором (admin@neftegaz.ru). В списке появляются кнопки
> «+ Добавить» и «Удалить». Создаю новое месторождение «Уренгойское»,
> отправляется POST-запрос на `/api/fields` — в DevTools во вкладке Network
> видно сам запрос, ответ и обновление таблицы.
>
> Открываю карточку месторождения — справа подгружаются связанные скважины
> (GET `/api/wells?field_id=…`). Перехожу в раздел «Скважины», фильтрую по
> статусу «operating». Создаю заявку на обслуживание скважины: указываю
> приоритет «high», она появляется во вкладке «Заявки». Меняю статус
> прямо из таблицы — улетает PATCH `/api/requests/{id}`.
>
> Открываю Swagger по адресу `/docs` — там описаны все эндпоинты с
> примерами тел запросов.
>
> Запускаю тесты: `pytest` — все ~9 тестов проходят успешно.»

### 3. Архитектура (≈1,5 мин)

> «Проект разбит на два сервиса.
>
> Backend — пакет `app` с четырьмя роутерами (`auth`, `fields`, `wells`,
> `requests`). Модели описаны в `models.py` через SQLAlchemy 2.0
> (декларативный стиль). Pydantic-схемы в `schemas.py` отвечают за
> валидацию запросов и ответов. Авторизация вынесена в `auth.py`
> (bcrypt + JWT) и `deps.py` (зависимости `get_current_user` и
> `require_admin`).
>
> Frontend — SPA на React Router. `auth.jsx` хранит контекст пользователя,
> токен лежит в localStorage. HTTP-клиент `api.js` оборачивает fetch и сам
> подставляет заголовок Authorization. Страницы (Fields, Wells, Requests)
> работают с соответствующими эндпоинтами.
>
> Миграции через Alembic — одна стартовая миграция `0001_initial`,
> создающая все таблицы и enum-типы. Тестовая БД — отдельная база,
> схема создаётся через `Base.metadata.create_all` на сессию pytest.»

## Возможные вопросы преподавателя

| Вопрос                                                            | Ответ                                                                                                 |
|-------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------|
| Где валидируются данные?                                          | На уровне Pydantic-схем (длины, диапазоны, email). И на уровне БД (NOT NULL, UNIQUE, FK).             |
| Как организовано хранение паролей?                                | Хеш bcrypt в поле `password_hash`, в API наружу не отдаётся.                                          |
| Как защищены приватные эндпоинты?                                 | Зависимость `get_current_user` парсит JWT; `require_admin` проверяет роль.                            |
| Что делает FK `ON DELETE CASCADE`?                                | При удалении месторождения автоматически удаляются связанные скважины и их заявки.                    |
| Что делает FK `ON DELETE SET NULL` у `author_id`?                 | При удалении пользователя его заявки не теряются, просто остаются без автора.                         |
| Как тесты не портят основную БД?                                  | Используется отдельная база `neftegaz_test` (порт 5433), таблицы пересоздаются на сессию pytest.      |
| Зачем нужен `vite proxy`?                                         | Чтобы фронт обращался к `/api/...` без CORS-проблем в разработке.                                     |
| Что будет, если изменить модель?                                  | Сгенерировать новую миграцию: `alembic revision --autogenerate -m "..."` и `alembic upgrade head`.    |
